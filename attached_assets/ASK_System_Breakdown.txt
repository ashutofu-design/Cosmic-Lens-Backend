1. QUESTION FLOW — "meri shaadi kab hogi?"
-------------------------------------------------------------------

Step-by-step trace of what actually executes:

(a) HTTP entry — flask_app.py:3942  POST /api/ask
    - Auth check (Firebase ID token if present). user_id cast to int.
    - Daily quota check (free vs paid tier).
    - Reads payload: question, lang, birth, kundli, history (last few turns).

(b) Brand-safety filter — openai_helper.py:937  _is_brand_unsafe(question)
    - Regex blocks any attempt to extract "are you AI / GPT / ChatGPT / which model".
    - Returns canned brand-safe response if flagged → AI never invoked.

(c) No-chart failsafe — openai_helper._no_chart_failsafe
    - If kundli is missing/empty → asks user to complete birth profile. AI not called.

(d) Topic classification — openai_helper.py:864  _classify_topic(question)
    - Pure regex/keyword matching, deterministic.
    - Scans question for tokens like shaadi|vivah|marriage|partner|7th house|kalatra|spouse|manglik|kundli milan.
    - Returns one of: marriage, career, health, wealth, education, child, litigation, travel, general.
    - For "meri shaadi kab hogi?" → topic = "marriage", confidence = 0.95.
    - This is NOT AI. It's a hand-crafted token-score map.

(e) Question language detection — openai_helper.py:581  _detect_question_lang(question, fallback)
    - Counts Devanagari chars → Hindi (hi).
    - Counts Hinglish tokens (mai/karu/jau/raha/lagta/hoga/etc, ~50+ words) → Hinglish (hn).
    - Else falls back to user's UI lang.
    - For "meri shaadi kab hogi?" → hn (Hinglish) because mostly Latin chars but >10% Hinglish tokens.

(f) Intelligence pre-computation — DETERMINISTIC
    Three modules run BEFORE the AI sees anything:
      1. chart_intelligence.analyze_chart(kundli, birth) — dignities, yogas, house-lords, Mangal-dosh, Sade-Sati.
      2. kp_engine.calculate_kp(birth) — KP cusps, sub-lords, planetary significators table.
      3. flask_app.current_transits() route's helper — current planet positions (live transits).
    These produce structured JSON facts. NO interpretation, just raw classical computations.

(g) Prompt assembly — openai_helper.py:625  _build_messages(...)
    The system message is hand-written and contains:
      - Persona ("Acharya Vidyasagar", warm, Hinglish if hn, Hindi if hi).
      - INTEL block (formatted facts from step f).
      - KP block (cusps + significators).
      - Transit block (where Jupiter/Saturn/Rahu currently are).
      - Topic FOCUS block — for marriage, this is the 5-step prioritized logic block from
        _TOPIC_FOCUS["marriage"] (170+ lines).
      - 11 strict rules (brevity 100-140 words, citations, no AI mention, marriage-specific
        3-paragraph format, etc).
      - History — last few turns from the user-session for context continuity.

(h) AI call — openai_helper.py:1009
    - model = "gpt-4o-mini" via Replit's OpenAI integration proxy.
    - temperature = 0.85, max_tokens = 480.
    - Single completion. No tool-use, no function-calling, no chain-of-thought, no second pass.

(i) Response parsing & return
    - text extracted. Topic + confidence + detected_lang attached. Returned to mobile as JSON.

4. DECISION ENGINE
-------------------------------------------------------------------

There is no numerical scoring system for marriage. There is no "if score > X then promise =
true". The logic is:

  1. Pre-compute facts deterministically (the 20 factors above where flagged "deterministic").
  2. Pour them into a tightly structured prompt with a 5-step prioritized FOCUS block
     instructing the AI HOW to weigh them.
  3. Let the AI synthesize.

Priority logic (enforced via prompt instructions, NOT code):
  - Step 1 (DENIAL):  If KP 7th-cusp sub-lord signifies only 1/6/10 → marriage denied/long-
                      delayed. Else → promised.
  - Step 2 (TIMING):  Marriage only in dasha/antardasha of planets signifying 2/7/11.
  - Step 3 (TRIGGER): Need Jupiter currently transiting/aspecting 1/5/7 for confident timing.
  - Step 4 (DELAY):   Saturn on 7th OR Mangal-dosh OR Sade-Sati → +1.5 to 2 years.
  - Step 5 (DOSHA):   Check cancellation if Mangal-dosh flagged.

Then prompt-enforced 3-para format: Pranam + Saptamesh / KP+Dasha evidence / Year-range +
remedy.

This priority is enforced linguistically, not algorithmically. The AI mostly follows it
(~85-90% of test runs) but can deviate.

5. TIMELINE LOGIC ("2026-2028")
-------------------------------------------------------------------

How the year-range gets produced:

  1. Dasha map is real. kundli.currentDasha has Maha + Antar with start/end dates → AI sees
     actual Gregorian dates.
  2. Antardasha sequence is implicit. AI knows the standard Vimshottari order (Ven→Sat→Merc
     →Ket→Sun…) and roughly when each Antar will run inside the current Maha.
  3. KP significator table tells AI which planets signify 2/7/11.
  4. Live Jupiter transit tells AI when Jupiter activates 1/5/7.
  5. AI then writes a year-range by intersecting (a) "next Antardasha of a 2/7/11
     significator" with (b) "Jupiter favourable transit window".

Brutal truth: The year-range is AI inference, NOT a deterministic calculation. The system
never computes "the next Antar of a 2/7/11 significator runs from Date X to Date Y, and
Jupiter aspects 7H during this window". It hands the data to the AI and trusts it to do
the intersection mentally.

This means:
  - Years are usually within 1-2 years of correct.
  - Sometimes the AI picks the right Antar but wrong year span.
  - Multiple consecutive runs of the same question can produce 2025-2027 vs 2026-2028 —
    non-deterministic.

7. WEAKNESSES — BRUTAL HONESTY
-------------------------------------------------------------------

A. Where the system is straight-up GUESSING

  1.  Year-range — AI inference, not computed. ±1-2 year drift between runs of same question.
  2.  Pratyantardasha (sub-sub period) — never computed. Without this, marriage timing
      precision below 6 months is impossible.
  3.  D9 Navamsa 7th lord — D9 chart IS computed in kundli_engine.py but never extracted
      into the INTEL block. The AI is told to consider it but has no actual D9 data → it
      fabricates plausible-sounding D9 statements ~50% of the time.
  4.  Jaimini Dara-karaka — not computed at all. AI fabricates.
  5.  Upapada Lagna — not computed at all. AI fabricates.
  6.  Shadbala — module exists (shadbala.py) but is not wired into INTEL. AI cannot say
      "Venus has X rupas" — it guesses descriptive strength.
  7.  Ashtakavarga & Sarvashtakavarga — not implemented. Cannot do bindu-based transit timing.
  8.  Spouse description (appearance, profession, region) — purely AI poetry. No deterministic
      basis (would need DK + UL + 7L sign + Navamsa).
  9.  Remedy selection — AI picks from a vague list in the prompt. Not tied to actual
      planetary weakness scoring.

B. Where logic is INCOMPLETE

  10. No verdict scoring layer. Industry-grade software (Jagannatha Hora, Parashara's Light)
      compute a numerical "marriage strength score" 0-100 BEFORE narrating. We don't.
  11. No KP "Ruling Planets" computation. KP marriage timing classically uses Ruling Planets
      at the time of the question — we only use natal+running-DBA significators.
  12. No nadi-amsa precision. Cannot answer ultra-fine spouse-trait questions classically.
  13. History context is dumped raw — no summarization. Long sessions push earlier turns out
      of the 480-token budget.
  14. Same question, different wording can land in different topics if the keywords miss
      (e.g. "biwi kab milegi?" might fall to general if "biwi" is not in the marriage token list).
  15. Multi-part questions ("shaadi kab + ladka/ladki kya hoga?") — the second sub-question
      for child gender is classically unanswerable from natal alone, but the AI sometimes
      still attempts an answer (we patched this in rule 0; ~80% honest now, 20% still guesses).

C. Where accuracy can FAIL

  16. Wrong birth time → cascading failure. Lagna, all house-positions, KP cusps, dasha
      balance ALL shift. The system does NOT do birth-time rectification. A 10-min error in
      birth time = different 7th cusp sub-lord = opposite verdict.
  17. Place geocoding — frontend geocodes; if the user types an ambiguous place, lat/lon can
      be wrong → ayanamsa offset still right but houses wrong.
  18. AI temperature 0.85 is high. Same question, 5 runs → 5 slightly different answers.
      Year-ranges vary, remedy varies, sometimes the verdict polarity flips on borderline charts.
  19. Token cap of 480 sometimes truncates the third paragraph mid-sentence on dense charts.
  20. AI sometimes overrules the FOCUS priority logic when the chart is ambiguous — e.g. it'll
      skip Step 1 (KP denial check) and jump to Step 3 if Jupiter looks dramatic.
  21. No sanity-check post-pass. Nothing reviews the AI output for internal contradictions
      ("strong yoga but marriage denied" can slip through).
  22. No confidence score on the verdict itself. We only have topic-classification confidence
      (0.95 for marriage detection). The actual answer's reliability is opaque to the user.

D. Honest accuracy estimate

  For the typical "shaadi kab hogi?" use case:
    - Topic detection:                                            99% correct.
    - Language detection:                                         95% correct (Hinglish edge-cases occasionally misroute).
    - Listing the right chart factors (7L, Venus, Mars, dasha):   95% correct because deterministic.
    - Direction of verdict (yes / delay / no):                    ~75-80% aligned with what a senior KP/Parashari astrologer would say on the same chart.
    - Year-range within ±1 year of "true" timing:                 ~55-65%.
    - Year-range within ±3 months:                                ~25-30% (because no Pratyantardasha).
    - Remedy classically appropriate to weakest significator:     ~70%.

  Path to 95% deterministic: Wire D9 7th-lord, Jaimini DK, Upapada Lagna, Shadbala,
  Ashtakavarga, Pratyantardasha, and a numerical 0-100 "marriage strength" + denial-flag
  computed in Python BEFORE the AI runs. Estimated effort: 2-3 focused sessions. Already
  partially scaffolded — shadbala.py and D9 chart math exist but unintegrated.

TL;DR
-------------------------------------------------------------------

  - Inputs to AI: ~75% deterministic Vedic + KP + transit math (real classical computations).
  - Decision/timing/interpretation: 100% AI inference (gpt-4o-mini, temp 0.85).
  - No scoring engine. No verdict pre-computation. No D9/DK/UL/Shadbala extraction.
  - Marriage verdict directionally correct ~75-80% of the time.
  - Year-range correct within ±1y ~60% of the time.
  - Major gaps are known and fixable; module skeletons already exist for several of them.
