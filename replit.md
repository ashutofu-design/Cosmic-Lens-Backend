# Cosmic Lens — Workspace

## Overview

Cosmic Lens is a pnpm workspace monorepo utilizing TypeScript for a mobile Vedic Astrology application. The project aims to deliver accurate and comprehensive astrological analysis, providing modern-context reframing and actionable insights. Key features include in-depth kundli calculations, numerology, Vastu analysis, and AI-driven astrological interpretations. The long-term goal is to achieve over 97% accuracy through a robust calculation engine, a sophisticated AI brain, and continuous learning.

## System Architecture

The project is a pnpm workspace monorepo using Node.js 24 and TypeScript 5.9.

**Mobile App (Expo React Native):**
- **Artifact**: `artifacts/cosmic-lens-mobile`
- **Navigation**: Bottom tabs for Home, Kundli, AI Chat, Insights, Notifications, Profile.
- **Features**: Daily Rashifal, Lucky elements, Panchang, Kundli Milan, Muhurat finder, Numerology, Remedies, Vastu tips, Dosha analysis, Forecast.
- **UI/UX**: `CosmicBg.tsx` for full-screen nebula background; `ThemeContext.cardShadow` for card glow; glassmorphism effect; zodiac-based accent theming; dark/light theme support.
- **Localization**: i18n support for 25 languages (12 Indian, 13 global) with region detection. Two-layer architecture for UI and Vedic content. Strict no-leakage rule between languages. Full UI and Vedic vocabulary localization complete, including RTL support for Arabic.
- **Risk Radar**: Integrated into the Forecast page as a comprehensive card, providing next-24-hours signals and 7-day outlook. Features a Vedic-themed UI with Sri Yantra Shatkona, 27 Nakshatra rim dots, Devanagari house markers, and an ॐ-centric center hub.

**Backend (Python Flask):**
- **Artifact**: `artifacts/api-server`
- **Core Components**: `flask_app.py`, `kundli_engine.py`, `kp_engine.py`, `ask_engine.py`, `requirements.txt`.
- **Architecture**:
    - **Modular Engines**: Vedic astrological calculations are organized into numerous specialized Python modules (e.g., `planet_strength.py`, `ashtakavarga.py`, `transits.py`, `divisional_charts.py`, `remedies.py`, `kp_cuspal_sub_lord.py`, `jaimini.py`, `chara_dasha.py`, `dosh_deep.py`, `dasha_extras.py`, `varga_deep.py`, `ashtaka_deep.py`, `transit_deep.py`, `kp_deep.py`, `special_lagnas.py`, `sahams_extended.py`, `nadi.py`, `lal_kitab_full.py`, `compat.py`, `muhurta.py`, `panchang.py`, `numerology.py`, `astro_vastu_engine.py`, `medical_engine.py`, `modern_context_engine.py`, `financial_engine.py`, `remedies_deep_engine.py`, `astrocartography_engine.py`, `timing_engine.py`).
    - **LOCKED FACTS Protocol**: `locked_facts.py` centralizes deterministic data from all engines for AI responses.
    - **AI Orchestration**: `openai_helper.py` injects `locked_facts_str` and applies "mirror rules" for verbatim citation.
    - **Deterministic Post-Injectors**: A safety layer in `ai_ask()` that ensures mandatory facts are cited.
    - **AI Brain Phase**: Future `ai_brain/` will house advanced AI components for system prompts, decision trees, RAG, and verification.
    - **RAG (Retrieval-Augmented Generation)**: Uses `chunker.py`, `embedder.py`, `retriever.py`, and `ingest.py` for classical Vedic knowledge from PostgreSQL `pgvector`, explicitly bypassed for timing questions.
    - **Life Mastery Report**: A comprehensive 17-tier PDF generation system combining numerology and Vedic astrology.
    - **Daily Energy Score**: A sophisticated system (`energy_engine.py`) that calculates a daily energy score based on transit-to-natal aspects, Jupiter & Mars house overlays, Choghadiya/Hora time-of-day, Tithi, and active Pratyantar-dasha lord transits. It includes a hard cap for Sade Sati Madhya, Mahadasha Sandhi detection, and active PD lord retrograde advisories. Scoring is based primarily on Moon-driven factors (75%) with Dasha providing a 25% backdrop. The system incorporates realistic scoring with lower bases and a compression curve for higher scores, seasonal sunrise/sunset calculations, degree-based aspects with orbs, and a Lagna + Chandra-lagna weighted blend for house calculations.
    - **Dosha Analysis**: `dosh_engine.py` expanded to 15 classical doshas including strict-classical rewrites for Putra Dosh, Pitru Dosh, Shrapit Yog, Daridra Yog, Guru Chandal, and Kaal Sarp Dosh (with 12 variants). Chandra-Mangal Dosh has been removed as a standalone entry. Kemadruma cancellation rules are strictly applied.
    - **Daily Lucky Engine** (`lucky_engine.py` + `/api/lucky/today`): Personalised "Aaj Ka Shubh Ank" + "Aaj Ka Shubh Rang" hero on Risk Radar. Computes from user's mool ank (digital root of birth date) + janma nakshatra cross-referenced with today's tara (nakshatra friendship), tithi, and weekday. Favourable taras (Sampat/Kshema/Sadhana/Mitra/Ati Mitra) amplify with day-lord planet color; inauspicious taras (Janma/Vipat/Pratyak/Naidhana) protect with mool-ank-planet color. Single hero number + single color (no fake 3-grid). Fail-loud: returns `{ok:false, error, message}` when birth data missing — NEVER fakes a value. Endpoint accepts both authenticated mode (`user_id` + `X-API-Key` → DB lookup) AND stateless mode (`{kundli, birthData}` in request body) so local-only kundlis (demo flow, pre-sync profiles) compute correctly. Mobile component `RiskRadarCard.tsx` reads local kundli/birthData from `UserContext` and forwards them. **Per-day extension (Apr 2026):** `compute_per_day_enrichment()` in `risk_text_engine.py` lazy-imports `compute_daily_lucky` and computes `shubh_ank` / `shubh_rang_name` / `shubh_rang_hex` for each of the 7 forecast days using projected Moon/Sun longitudes + that day's ISO date. Threaded through `enrich_risk_radar(... birth_data, today_date_iso)` and the `/api/risk-radar` route. Mobile reads them via `PerDayRisk` and renders via a derived `selLucky` keyed on `selOffset === 0` (Day 0 → richer `/api/lucky/today` payload with reasoning; Day 1-6 → per-day fields, label drops the "AAJ KA" prefix). Missing inputs (no birth data / lucky engine off) propagate as `null` and the UI shows an explicit "abhi available nahi" fallback — NO fake values.
    - **Cosmic Intelligence Text Layer** (`risk_text_ai.py`): The 5-field guidance text (kya_risk_hai / kya_dhyan_rakhna_hai / kya_avoid_karna_hai / kya_karna_hai / upay) + category label for every Risk Radar trigger is now **AI-generated friendly Hinglish** rather than hand-written templates. `get_text(trigger)` calls OpenAI (`gpt-4o-mini` via Replit AI Integrations proxy → `AI_INTEGRATIONS_OPENAI_BASE_URL` + `AI_INTEGRATIONS_OPENAI_API_KEY`; falls back to `OPENAI_API_KEY` direct). Process-wide in-process cache keyed on trigger (12 unique triggers max) → first hit per trigger generates once (~1-3s), all subsequent hits are O(1). Background prewarm thread auto-fires on module import (`prewarm_async()`) so most production users never wait for a generation. `_TRIGGER_BRIEFS` carries (a) internal `brief` describing what cosmic situation the trigger represents and (b) `upay_hint` telling the model the canonical deity/mantra/daan for that signal — keeps remedies aligned (Chandrashtama→Chandra mantra, Mars→Hanuman Chalisa, Saturn→Shani Dev, etc.) instead of letting the model invent mismatched remedies. STRICT brand voice in `_SYSTEM_PROMPT`: never reveals AI/GPT/LLM/model/algorithm; never mentions raw planet names (Mars/Saturn/Mercury) or technical jargon (dasha/nakshatra/tithi/rahukaal) in output text — uses generic conversational terms (`aaj ki energy`, `cosmic vibration`, `sensitive samay window`); EXCEPTION: deity names + mantras + traditional remedies ARE allowed in the `upay` field (user-expected). Engineering fallback: `risk_text_engine._resolve_text(trigger)` tries AI first, falls through to the deterministic `_TEXT_MAP` (hand-written Hinglish — NOT random/fake) only on AI unavailability so the UI never breaks. Both `compute_per_day_enrichment()` (per-day loop) and `enrich_risk_radar()` (top_risk) use `_resolve_text()`.
    - **Risk Text Engine** (`risk_text_engine.py` + `/api/risk-radar` enrichment): Replaces hardcoded mobile RISK_BY_LEVEL templates and BEST_TIME_SLOTS arrays with real engine output for ALL 7 days (not just today). `detect_dominant_trigger()` mirrors `compute_risk_radar`'s severity ordering (Volatile=10, Chandrashtama=9, Tara Naidhana=8, Saturn Heavy=8, Mars Active=7, Tara Mild=6, PD Weak=6, Tithi Rikta=6 [precedence over Amavasya], Amavasya=6, Saturn Mild=5, Rahukal=4) over 12 trigger keys. Each trigger maps to a 5-field Hinglish text bundle (kya_risk_hai / kya_dhyan_rakhna_hai / kya_avoid_karna_hai / kya_karna_hai / upay) + category label. `build_choghadiya_schedule()` produces 16 day+night segments with quality (best/neutral/avoid) reusing energy_engine's `_DAY_CHOGHADIYA`/`_NIGHT_CHOGHADIYA`/`_RAHUKAL_SEGMENT`. `pick_best_avoid_times()` selects the next-upcoming Amrit/Shubh/Labh as BEST and Rahukaal-or-next-Rog/Kaal/Udveg as AVOID. Sunrise/sunset uses `_astronomical_sun_times` (NOAA SPA when birth lat/lon present) or `_seasonal_sun_times` fallback. **Per-day projection** (`compute_per_day_enrichment()`): for each of the next 7 days the engine projects today's Moon longitude forward by `i × 13.176°`, Sun by `i × 0.9856°`, advances weekday `(today_weekday + i) % 7`, recomputes Tara Bal vs natal nakshatra, recomputes Tithi (Rikta/Amavasya) from projected Sun-Moon difference, rebuilds Choghadiya schedule + best/avoid windows for that day's weekday, and carries forward persistent slow-mover signals (Saturn phase, Mars affliction, PD weakness) — Day-0-only signals (volatile_day, chandrashtama) attach only to today. The Flask endpoint always emits `enriched: true|false` flag so the client can detect partial failures. Mobile `lib/riskTextAPI.ts` exposes `PerDayRisk` type + `per_day?: PerDayRisk[]` on the response. `RiskRadarCard.tsx` computes `selOffset = days[selected].date - today` (calendar days, local), looks up `riskApi.per_day[selOffset]`, and overrides selData with the per-day fields when `apiOk = enriched && perDay && perDay.best_time.window && perDay.avoid_time.window` — covering Day 1 through Day 7. The risk-level gauge color/label/marker, KYA RISK / DHYAN / AVOID / KARNA / UPAY sections, BEST TIME and AVOID TIME tiles, and the "✨ Powered by Advanced Cosmic Intelligence" footer all render from real per-day engine output for every covered day. Outside the 0..6 offset window (e.g. far-future demo days) the deterministic local template still applies but the powered-by badge is suppressed. STRICT no-template-fallback gate persists: when API fails or enrichment is incomplete, the card shows "Cosmic signals abhi fetch nahi ho paaye" with "—" placeholders — never the hardcoded RISK_BY_LEVEL template.
