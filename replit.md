# Cosmic Lens
A mobile Vedic Astrology application delivering highly accurate astrological analyses, including kundli calculations, numerology, Vastu, and AI-driven interpretations.

## Run & Operate
- **Run**: `pnpm dev`
- **Build**: `pnpm build`
- **Typecheck**: `pnpm typecheck`
- **Codegen**: `pnpm codegen`
- **DB Push**: `pnpm db:push`
- **Required Env Vars**: `OPENAI_API_KEY`, `DATABASE_URL`, `HEALTH_LLM_MODEL`, `HEALTH_ROUTER_MODEL`, `FINANCE_STATIC_BYPASS`, `HEALTH_STATIC_BYPASS`, `LEAN_PACK_MODE`, `LAYER2_FUZZY`, `LAYER3_LLM`

## Architecture decisions
- **Monorepo Structure**: Uses pnpm workspaces for dependency management and code sharing.
- **CLE (Cosmic Lens Engine) Format**: Backend astrological calculations follow a strict modular engine format.
- **LOCKED FACTS Protocol**: Deterministic data for AI responses is centralized in `locked_facts.py` to prevent hallucination.
- **AI Orchestration**: `openai_helper.py` manages fact injection, verbatim citation, and deterministic post-injectors.
- **Two-Layer Prompting**: "Guided Freedom" approach for LLM prompts (80% mindset, 20% surgical guards).
- **Engine-Controlled Verdicts**: For complex questions (e.g., marriage timing), the final verdict is deterministically generated by the engine, not the LLM.
- **Signal-Pack Layer**: Engine builds a compact JSON signal-pack with normalized states for LLM expression, preventing raw astrological data exposure.
- **Universal Signal Pack v2**: Replaced per-question-type registries with one universal pack containing all four health dimensions for LLM interpretation.
- **Mandatory Final Verdict**: Enforces a sharp, engine-built final verdict line (`👉 Final:`) in responses, stripping abstract metaphors and ensuring conciseness.
- **Property Static Engine (P1.0)**: A static engine (sister to health_static) for property-related questions, using a signal-pack and strict output formatting. Currently DISABLED by default, routing property Qs to the LLM with full chart data.
- **KP-ALWAYS-FULL**: When enabled, sends all 12 cusps and all 9 planet significators to the LLM for every question, regardless of topic, and preserves it through narrative-mode trim.
- **Property Focus Block (P1.1 → P1.2 → P1.2.3 INTENT-split, 2026-05-06)**: Per-topic analysis framework injected into LLM system prompt for property/ghar Qs. **P1.2.3 fix (user-feedback)**: TIMING block was "always-append" causing dasha forecasting to leak into pure STATIC/QUALITY questions ("property yog hai?", "delay hoga ya early?"). Split INTENT axis into 3 distinct atomic blocks: `STATIC_YOG` (yog existence + strength + houses, NO dasha/transit), `YOG_QUALITY` (yog NATURE — slow/fast/smooth/friction maturation factors, NO "kab milega"), `TIMING` (gated on explicit triggers `kab/when/abhi/turant/next month/year/this year/upcoming/near future/muhurat`). "Delay" and "early" now correctly route to YOG_QUALITY (nature words) not TIMING. Framework header + 16 worked examples re-categorized into STATIC / QUALITY / TIMING / ACTION-SCOPE buckets. Answer-style guards added: STATIC_YOG/YOG_QUALITY answers must NOT use phrases like "near term movement", "this phase me", "abhi chal raha hai" — those are TIMING-only. Validation 9/9 explicit-phrase guards on P40 chart (3 STATIC + 3 QUALITY + 3 TIMING). Original failing Q ("property me delay hoga ya early yog he?") now answers via YOG_QUALITY with maturation framing ("delayed-matured", "late-settling", "responsibility ke baad solid") instead of Jupiter-Rahu dasha forecasting. **P1.2 architecture replaces P1.1 fat constant with composable atomic blocks** — property has ~1000 possible Qs across 4 axes (ACTION × SCOPE × INTENT × ASSET = 720+ combos), so enumeration doesn't scale. Solution: ~14 atomic CHECK blocks (~150 chars each) in new module `artifacts/api-server/property_focus_routing.py` (`build_property_focus(question)` returns ~5.3 KB framework header + atomic-blocks dump + answer-style rules). LLM detects axes from user Q and self-routes to 2-4 relevant atomic blocks. Atomic blocks: `BUY / SELL / INHERIT / RENT / BUILD / DISPUTE / ANALYZE` (action) + `FOREIGN / LAND / COMMERCIAL / MULTIPLE` (scope modifiers) + `TIMING / RISK / REMEDY / REFUSE_TIMING` (always-append). 12 worked examples in header calibrate routing. **Wired at all 4 LLM call sites for parity** (since `LLM_FULL_CHART_MODE` defaults ON → passthrough is dominant path): narrative `_build_messages` (~L4448 elif), sync passthrough (~L14346), stream passthrough (~L17934), legacy passthrough (~L3794). Doctrinal hedges preserved: vargottama (supportive ≠ guarantee), neecha-bhanga (requires cancellation conditions), KP 4/11/12 (core is 4 + (11 OR 2)). Killswitch `PROPERTY_FOCUS_BLOCK ∈ {0,false,no,off}`, default ON. **P1.2.1/.2 hardening (architect feedback)**: regex split into STRONG (always-fire on unambiguous tokens like ghar/property/paitric/construction/renovation/kabza) vs WEAK (ambiguous tokens like land/shop/office/sell/build) requiring an ANCHOR co-occurrence (transaction verbs `kharid\w*|bech\w*|banwa\w*` + ownership phrases EMI/loan/joint/biwi-ke-naam + sampatti/invest). `_NON_PROPERTY_TOPIC_IDS` frozenset gates regex when topic_id is explicitly career/finance/health/marriage/etc. — only STRONG hits override. Module-level `_PROPERTY_FOCUS_FALLBACK_COUNT` counter incremented at each call site when import/exec of `build_property_focus` fails (sustained non-zero growth = real regression to alert on). 2 NEW edge atomic blocks added: `JOINT_TITLE` (joint ownership / biwi-ke-naam → 7H+7L+4H combined) and `LOAN_EMI` (home loan/EMI → 6H/6L debt + 11H gain capacity + dasha-lord 6/8/12 burden risk). 2 deterministic doctrinal lines softened: LAND ("water signs = quality concern, NOT hard veto"), TIMING ("Saturn/Jupiter transit = supportive/indicative window, NOT guarantee"). **Validation post-hardening**: 40-Q coverage 39/40 (only "disposal ka yog hai" misses, genuinely ambiguous), false-positive guard on 15 non-property Qs (including "office politics", "sell mutual fund", "build career", "rental car milega") = **0/15 leaks**. 8 real LLM smoke tests on P40 chart with new edge atomics: **8/8 signal-match** including JOINT_TITLE ("joint title, khas kar spouse ke naam... formal tareeke se secure") and LOAN_EMI ("6th house jo loan aur debt dikhata hai, uska lord Venus 1st house me"). Pairs with P1.0-disable + KP-ALWAYS-FULL.

## Gotchas
- **Static Gate Parity**: Any new engine/gate added to `/api/ask` MUST simultaneously be added to `/api/ask/stream`.
- **Cache Invalidation**: Any policy change requires bumping the cache namespace (`_vX`).
- **Vedic Vocab Stripping**: The validator strips Vedic terms unless explicitly opted in or the user's question contains a "tech-request" keyword.
- **Topic-Lock Conflicts**: When multiple injection points compete, the latest injected point in the user message order wins.
- **DB-Load Enforcement**: For authenticated users, `kundli` is always loaded from the database; client-supplied `kundli` is for anonymous demo mode only.
- **LLM Temperature**: The `temperature` parameter is often rejected by `gpt-5` models, meaning LLM strict mode is currently dormant for those models.
- **Telemetry Best-Effort**: Telemetry inserts are wrapped in `try/except` and silently drop rows on contention.
